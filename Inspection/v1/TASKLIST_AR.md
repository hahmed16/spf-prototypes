# قائمة مهام تطبيق الملاحظات - Inspection/v1

تاريخ الإنشاء: 2026-04-23  
النطاق: `Inspection/v1`  
الحالة العامة: قيد المتابعة

## 1) حالات البلاغات (Workflow Statuses)

- [ ] اعتماد قائمة الحالات التالية بالنص العربي الموحد داخل النظام:
  - [ ] مسودة
  - [ ] تم تقديم البلاغ
  - [ ] تم اعادة فتح البلاغ
  - [ ] قيد المراجعة
  - [ ] تم رفض البلاغ
  - [ ] بانتظار اعتماد رئيس قسم المتابعة والبلاغات
  - [ ] تم توجيه البلاغ للتفتيش
  - [ ] قيد المراجعة من قسم التفتيش
  - [ ] بانتظار اجراء الزيارة التفتيشية
  - [ ] بانتظار اعتماد رئيس قسم التفتيش
  - [ ] انتظار اعتماد مدير الدائرة
  - [ ] تم اغلاق البلاغ
  - [ ] تم حفظ البلاغ
- [ ] ربط كل حالة بالحدث الصحيح عند انتقال البلاغ بين الأدوار.
- [ ] التحقق أن الحالات تظهر بنفس التسمية في:
  - [ ] قائمة البلاغات
  - [ ] تفاصيل البلاغ
  - [ ] الـ Action Panel
  - [ ] سجل الإجراءات (History/Timeline)

## 2) قواعد عامة لسير العمل

- [ ] تطبيق قواعد إظهار البلاغات حسب الدور (الشفافية) كما يلي:
  - [ ] مستخدم الصندوق الداخلي: يرى بلاغاته المقدمة فقط.
  - [ ] المؤمن عليه: يرى بلاغاته + أي بلاغ يكون طرفًا فيه.
  - [ ] مستخدم الشركة الخارجي: يرى بلاغات الشركة + أي بلاغ تكون الشركة طرفًا فيه.
  - [ ] موظف قسم المتابعة والبلاغات: يرى البلاغات التي يبحثها بنفسه أو المعيّنة له.
  - [ ] رئيس قسم المتابعة والبلاغات: يرى كل البلاغات، والافتراضي بلاغات قسم المتابعة مع إمكانية البحث للاطلاع.
  - [ ] موظف قسم التفتيش: يرى البلاغات التي يبحثها بنفسه أو المعيّنة له.
  - [ ] رئيس قسم التفتيش: يرى كل البلاغات، والافتراضي بلاغات قسم التفتيش مع إمكانية البحث للاطلاع.
  - [ ] مدير دائرة التفتيش: يرى جميع البلاغات في جميع المراحل والحالات.
- [ ] ضبط انتقالات الإجراءات بحيث:
  - [ ] بعض الإجراءات تغيّر الحالة.
  - [ ] بعض الإجراءات لا تغيّر الحالة.
  - [ ] التوثيق واضح لكل إجراء هل يغيّر الحالة أم لا.
- [ ] تطبيق مبدأ العرض التراكمي للبيانات بين الأدوار:
  - [ ] كل دور يرى بيانات من سبقه + إجراءاته في Panel مستقل.
  - [ ] عند التصعيد/التوجيه تظهر Panels تاريخية منفصلة لكل دور.
- [ ] ضمان توفر حالة/طلب واحد على الأقل لكل دور يظهر فيه Action Panel فعليًا للاختبار.
- [ ] إلزام سبب إجباري عند الإجراءات التالية:
  - [ ] رفض
  - [ ] إغلاق
  - [ ] حفظ
  - [ ] إعادة فتح

## 3) الإجراءات المتاحة لكل دور

### 3.1 مقدم الطلب
- [ ] حفظ كمسودة
- [ ] حفظ وإرسال الطلب
- [ ] إعادة إرسال بعد طلب استيفاء
- [ ] دعم سيناريو استيفاء البيانات من أحد أطراف البلاغ (المؤمن عليه/الشركة)

### 3.2 موظف قسم المتابعة والبلاغات
- [ ] طلب استيفاء البيانات من كافة الأطراف
- [ ] طلب استيفاء البيانات من صاحب العمل فقط
- [ ] طلب استيفاء البيانات من المؤمن عليه فقط
- [ ] توصية برفض البلاغ
- [ ] توصية بقبول البلاغ
- [ ] توصية بحفظ البلاغ

### 3.3 رئيس قسم المتابعة والبلاغات
- [ ] إعادة تعيين الموظف المختص ببحث البلاغ
- [ ] إعادة البلاغ إلى الموظف
- [ ] تأكيد رفض البلاغ
- [ ] إغلاق البلاغ
- [ ] حفظ البلاغ
- [ ] توجيه لقسم التفتيش

### 3.4 موظف قسم التفتيش
- [ ] إعادة البلاغ إلى رئيس قسم المتابعة والبلاغات
- [ ] توجيه البلاغ إلى رئيس القسم

### 3.5 رئيس قسم التفتيش
- [ ] إعادة تعيين الموظف المختص ببحث البلاغ
- [ ] إعادة البلاغ إلى الموظف
- [ ] رفض إجراء التفتيش على البلاغ
- [ ] إغلاق البلاغ
- [ ] توجيه لقسم التفتيش

### 3.6 مدير الدائرة
- [ ] إعادة تعيين الموظف المختص ببحث البلاغ
- [ ] إعادة البلاغ إلى موظف قسم المتابعة والبلاغات
- [ ] إعادة البلاغ إلى موظف قسم التفتيش
- [ ] إعادة البلاغ إلى رئيس قسم المتابعة والبلاغات
- [ ] إعادة البلاغ إلى رئيس قسم التفتيش
- [ ] إغلاق البلاغ
- [ ] حفظ البلاغ
- [ ] إعادة فتح البلاغ

## 4) شاشة قائمة البلاغات (للأدوار الخارجية + مستخدم الصندوق الداخلي)

- [ ] تثبيت معايير البحث التالية فقط:
  - [ ] رقم البلاغ
  - [ ] الرقم المدني للمؤمن عليه
  - [ ] رقم السجل التجاري لصاحب العمل
  - [ ] من تاريخ
  - [ ] إلى تاريخ
  - [ ] حالة البلاغ
- [ ] حذف حقل "الموعد" من جدول البلاغات.
- [ ] استبدال "الموعد" بـ "تاريخ تقديم الطلب".
- [ ] توفير حالة بلاغ مناسبة لاستيفاء بيانات.
- [ ] عند عرض التفاصيل يظهر نص الملاحظة التي أعاد بها الموظف الطلب مع البيانات المطلوب استيفاؤها.

## 5) شاشة تفاصيل البلاغ

- [ ] استبدال مصطلح "دعوى عمالية" بـ "شكوى عمالية".
- [ ] إزالة أي طلب ملف باسم "خطاب صاحب العمل معتمد".
- [ ] إزالة مصطلح "المعني" من عبارة "بيانات صاحب العمل المعني".
- [ ] عند تقديم الطلب من المؤمن عليه:
  - [ ] خيار: صاحب العمل الحالي / صاحب عمل آخر.
  - [ ] عند اختيار صاحب العمل الحالي: تعبئة تلقائية لبيانات صاحب العمل.
  - [ ] عند اختيار صاحب عمل آخر: إظهار بحث برقم سجل تجاري + زر استعلام.
  - [ ] عند الاستعلام: عرض بيانات افتراضية.
- [ ] منع طلب المستندات التالية نهائيًا:
  - [ ] صورة الهوية الوطنية للعامل
  - [ ] مستخرج من نظام حماية الأجور
- [ ] توحيد قاعدة البيانات: إذا كان مقدم الطلب هو المؤمن عليه، فبيانات مقدم البلاغ = بيانات العامل.

## 6) مهام تحقق (QA/UAT) لضمان تغطية كل البنود

- [ ] اختبار كل حالة من حالات البلاغ عبر سيناريو فعلي.
- [ ] اختبار صلاحيات الإظهار لكل دور مستخدم.
- [ ] اختبار جميع الإجراءات المتاحة لكل دور ومنع الإجراءات غير المسموحة.
- [ ] اختبار إلزام سبب الإجراء (رفض/إغلاق/حفظ/إعادة فتح).
- [ ] اختبار العرض التراكمي للـ Panels في تفاصيل البلاغ.
- [ ] اختبار سيناريو استيفاء البيانات وظهور الملاحظة.
- [ ] اختبار مصطلحات الواجهة بعد التحديث.
- [ ] اختبار حقول البحث وقيم الجدول بعد حذف "الموعد".

## 7) تعريف الاكتمال (Definition of Done)

- [ ] كل البنود أعلاه مكتملة ومعلمة.
- [ ] لا توجد حالة أو إجراء مفقود مقارنة بالملاحظات الأصلية.
- [ ] تمت مراجعة نهائية على جميع الأدوار داخل `Inspection/v1`.
